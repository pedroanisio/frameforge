Gating against same-layer text collisions

The model is now settled: collision = an unintended overlap — overlap itself is a first-class effect, gated by consent, not legibility. What's left to decide is how to enforce it, given that a static box check provably floods and the only reliable ink lives at render time.

DECISION collision-gate/2026-07 HORIZON core CI infra · multi-year STATUS open · analysis TRIGGER amazon-proxy-2026 p1 header overlap
Rev 2 · the model is settled

Rev 1 asked where a text-collision gate can live and treated collision as a legibility problem. The design decision settled it as consent: collision is an unintended overlap; overlap is a first-class effect (watermarks, captions over images, double-exposure type). Full model in docs/spec/viewport-definition-proposal.md. Four consequences reprice this analysis:

  • Opacity is out — the check is opacity-blind volume geometry plus an authored flag (rev 1's occlusion-aware "Rule B" is superseded).
  • Cross-layer overlap is exempt by construction — collision is a 3D-AABB intersection (same x,y and same z-slab); different layers never collide. A principled exemption replaces rev 1's fuzzy opacity/role heuristic.
  • New prerequisite P0 — the model needs an overlap: allowed attribute (default false, mutual). A schema change, not a validator tweak.
  • Rev 1's "O3 authoring gate" is absorbed into P0 — the field is the authoring mechanism.

Unchanged & reconfirmed: box-overlap floods → reliable overlap needs rendered ink → ink needs layout + metrics → the check is render-time.

01

The decision's coordinate system

Before options: the constraints that prune the space, the horizon that sets reversibility cost, and the axes success is measured on.

Binding constraints — prune the space

  • B1The static validator has no fonts and no wrap engine, and must not import the render package (module-shadow). Proven to flood — see §02.
  • B2make check must be deterministic across machines. The golden + overflow gates deliberately run estimate mode for this reason.
  • B3Real glyph advances need fontTools + resolvable fonts — reproducible only in the font-rich frameforge image today.
  • B4PALS's Law — estimate/CV output is unverified; the gate must be honest about what it actually verified.

Soft constraints — rank, don't eliminate

  • S1A gate that only WARNs is ignored; hard-fail preferred where it can be made deterministic.
  • S2Minimize coupling of make check to Docker.
  • S3Reuse existing machinery: render_text's real geometry; --check-overflow telemetry.
  • S4Teach the fix (authoring guidance), not just flag pixels.

Success axes — how each option is scored

Detection reliability (low false-negative) · False-positive safety (a gate that floods gets disabled) · Determinism (identical verdict everywhere) · Enforcement strength (hard-fail > advisory) · Implementation cost · Decoupling (from Docker / renderer internals) · Authoring guidance.

02

The finding that reframes the decision

The user's original steer — "add a validator rule" — assumes a static geometric check is feasible. It is not. I ran four variants against the full committed fixture corpus. Every box-based approach floods, because a FrameGraph box is a layout region, routinely larger than the text it holds; box overlap ≠ text collision.

ApproachMetricsFalse positivesWhy it fails
Raw box overlap, any two same-layer textnone1090boxes are regions, bigger than their text
Box overlap + ink-separation suppressorestimate775blind to wrap & real glyph width
Per-line ink-box overlap, both axesestimate617single-line estimate over-reaches wrapped text
Overlap within a detected table gridnone1090+150-wide cells on 130px centres overlap by design

The residual false positives all trace to three things a static check cannot see: real glyph widths (uppercase advances ≈ 0.60, not the fallback 0.52 — the proxy's actual cause), text wrap, and legitimately loose or stacked author boxes (Age: above 62).

The proxy defect, measured

Header boxes [330,232,160,24] "Board's Voting Recommendation" and [462,232,90,24] "More Information" overlap by 28px as authored geometry. Under the renderer's own estimate (0.52×), the ink sits ~7px clear — so estimate-mode make check sees no problem. Under real Ember uppercase advances (~0.62×) the glyphs collide. The overlap you saw is the estimate-layout / real-render gap. Only render_text — which computes widest, top, total_h, anchor with wrap and real metrics — can see it.

Composition

O1, O3, and O7 are orthogonal in mechanism (pixel-detect vs. structural vs. metrics-plumbing) yet compose: layers of defense, not rivals. O7 is the enabling substrate — it upgrades O1's advisory count and O4's advisory tier into deterministic hard-fails without O2's Docker coupling. The only real conflict is O2-alone (hard-fail without O7 ⇒ local/CI divergence) and O6 (defer ⇒ tension with the fix-root-causes rule unless O7 is scheduled).

07

Recommendation — the decision function

Recommended path

P0 + O1 now · O7 as the funded follow-up

Ship P0 + O1 + the fixture fix as one move, and schedule O7 as the upgrade that later promotes the advisory count to a deterministic hard-fail everywhere.

▸ NOW — model + advisory detector, zero false-fail risk

P0 adds overlap: allowed to the model (default false, mutual) · O1 flags same-layer ink overlaps lacking unanimous consent as an advisory --check-overflow count under real metrics · the proxy is fixed by geometry (an accident, not an effect) and the golden re-pinned. Nothing here can false-fail the build.

▸ NEXT — dissolve the determinism blocker

O7 vendors a vetted metrics table (gated on the §below question). Once real measurement is reproducible off-Docker, flip O1's advisory count to a hard-fail — full enforcement, no container coupling. This is the path O2 wishes it were.

▸ AVOID

O2 alone (Docker-coupled hard-fail ⇒ local/CI split) and any gate before P0 (with no overlap: allowed, every intentional same-layer overlap is a false positive).

If your priorities differ

If you valueshipping now with no risk of blocking a build on font-dependent output
→ P0 + O1 + fix (the recommendation). Declare intent, enforce advisorily on rendered ink.
If you valueone deterministic hard-fail everywhere and will fund the metrics work
→ P0 + O7, then O1-hard / O4-hard. Highest rigor; the metrics table must be vetted against real renders first.
If you valuelanding the model but not the gate yet
→ P0 + fixture fix (O6). Acceptable only if O7 or O1 is actually scheduled — else the accident class stays undetected.
If make checkalready runs in-container for other gates
→ O2 becomes cheap. Its coupling cost is already paid; go straight to hard-fail.
08

What to resolve next

  • Fixes the model's cruxIs z-slab = the page.layers[] container (vs. a continuous per-object z)? This decides what counts as a same-layer intersection — the one open question in the definition doc.
  • Gates O7Can the fixture faces' advances be vendored faithfully? Render one fixture in-container (real metrics) vs. a candidate pinned table and diff the collision verdicts. If they disagree, O7 is not yet safe.
  • Sizes the P0 retrofitHow many same-layer ink overlaps survive in the corpus once the ink refinement and cross-layer exemption are applied? That is the true count of docs needing overlap: allowed (rev 1's 617 is an estimate-mode upper bound).
  • Scopes the detectorConfirm detection is limited to top-level layer objects (excluding Table/flow cells, which their own engines lay out without overlap). A one-line guard in render_text's caller.

Deep-dives available on request: spec the overlap: allowed field (schema shape, mutual-declaration semantics) · stress-test O7 against Chromium fallback-chain shaping · map the O1→O1-hard migration once O7 lands · triage the corpus same-layer ink overlaps into accident-vs-effect.
